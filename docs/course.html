01

От файла к библиотеке свойств

Что происходит, когда вы перетаскиваете IDS-файл в приложение

Что это за приложение?

IDS2PSET — это веб-приложение, которое превращает IDS-файлы (спецификации требований) в IFC-файлы (библиотеки свойств). Работает полностью в браузере — без сервера, без установки.

💡
Зачем это нужно?

Когда BIM-моделист создаёт здание в Revit или ArchiCAD, ему нужно добавить свойства к элементам: «кадастровый номер», «тип покрытия», «масса». IDS-файл описывает, какие свойства нужны. IDS2PSET превращает это описание в готовую IFC-библиотеку, которую можно импортировать в BIM-программу.

  for (const file of fileList) {
    if (!file.name.endsWith('.ids')) continue;
    const content = await file.text();

      .pyodideBridge.parseIDS(content);
    this.mergePSets(result, file.name);

  this.renderFilesList();

Как мост соединяет два мира

Pyodide: Python в браузере через WebAssembly

Мост через реку

Представьте два города, разделённых рекой. Один — JavaScript (язык браузера), другой — Python (язык для обработки IFC). Между ними нужен мост.

Pyodide Bridge — это этот мост. Он позволяет JavaScript-коду вызывать Python-функции и получать результаты обратно. Но мост не бесплатный: нужно сначала его построить (загрузить Pyodide), и перевозка грузов занимает время.

🌉
Почему Python в браузере?

Библиотека IfcOpenShell, которая создаёт IFC-файлы, доступна только на Python. Поэтому весь Python-стек приходится запускать в браузере через Pyodide.

Что загружается при первом визите

1

Pyodide (~30 МБ)

Python 3.13, скомпилированный в WebAssembly. Загружается с CDN при первом открытии.

2

micropip

Менеджер пакетов Python для браузера. Позволяет устанавливать Python-библиотеки.

3

ifcopenshell (wheel)

Специально скомпилированная версия IfcOpenShell для Pyodide. ~15 МБ.

4

Python-модули (src/)

ids_parser.py, pset_generator.py — копируются в виртуальную файловую систему Pyodide.

Чтение спецификаций

Как IDS Parser извлекает данные из XML

Что внутри IDS-файла?

IDS-файл — это XML-документ со структурой XML. Он содержит «спецификации» — правила, которым должна соответствовать BIM-модель.

XML (упрощённый пример) 
<ids:specification name="Требования к ПЗУ">
  <ids:applicability>
    <ids:entity>
      <ids:name>
        <xs:restriction base="xs:string">
          <xs:pattern value="IFCSTAIRFLIGHT|IFCSLAB"/>
        </xs:restriction>
      </ids:name>
    </ids:entity>
  </ids:applicability>

  <ids:requirements>
    <ids:property dataType="IFCTEXT">
      <ids:propertySet>
        <ids:simpleValue>МОГЭ_Информация</ids:simpleValue>
      </ids:propertySet>
      <ids:baseName>
        <ids:simpleValue>Наименование</ids:simpleValue>
      </ids:baseName>
    </ids:property>
  </ids:requirements>
</ids:specification>
📋
Структура спецификации

Каждая <ids:specification> содержит две части: applicability (к каким элементам применяется) и requirements (какие свойства нужны). Parser читает обе части.

Рождение IFC-файла

Как PSet Generator создаёт IFC4 через ifcopenshell

Что такое IFC-файл?

IFC (Industry Foundation Classes) — это открытый формат для BIM-данных. Представьте его как «универсальный язык» для строительных программ: Revit, ArchiCAD, Tekla — все понимают IFC.

Наш генератор создаёт IFC4 — четвёртую версию стандарта. Внутри файл содержит IfcPropertySetTemplate — шаблоны наборов свойств.

🏗️
Аналогия

IFC-файл — это как бланк анкеты. Он определяет, какие поля нужно заполнить (имя, дата рождения, адрес), но не содержит самих данных. BIM-моделист потом заполняет эти поля для каждого элемента здания.

Как создаётся IFC-файл

CODE (pset_generator.py) 
def generate(self, psets):
    self.create_library("IDS2PSET Library")
    for name, pset_data in psets.items():
        properties = []
        for prop in pset_data["properties"]:
            if prop.get("is_pattern"):
                continue  # Пропускаем regex
            ifc_prop = IFCTemplateProperty(
                name=prop["name"],
                data_type=prop["data_type"],
                enum_values=prop["enum_values"]
            )
            properties.append(ifc_prop)
        template = IFCTemplate(
            name=name,
            properties=properties,
            applicable_entities=pset_data["entities"]
        )
        self.add_template(template)
    return self.file.to_string()

Почему некоторые свойства не попадают в IFC

IDS-файл может содержать свойства с регулярными выражениями (regex). IFC-формат не поддерживает regex в определениях свойств, поэтому генератор их пропускает.

🚫
Жёсткий regex (xs:pattern)

Если свойство определено через <xs:pattern> в IDS — оно не попадёт в IFC. Приложение покажет предупреждение: «N свойств описаны регулярными выражениями — не будут созданы».

ℹ️
Мягкий regex (simpleValue)

Если regex указан в <ids:simpleValue> (без xs:pattern) — свойство попадёт в IFC, но приложение предупредит: «Возможно, IDS некорректен». Это может быть ошибка в IDS-файле.

Когда всё идёт не по плану

Отладка, предупреждения и общая архитектура

Типы предупреждений

Приложение показывает несколько типов предупреждений. Каждый — сигнал о потенциальной проблеме:

⚠️
Regex-свойства (оранжевый)
Свойства с регулярными выражениями не попадут в IFC. Нормальное поведение.
🔍
IDS некорректен (жёлтый)
Regex указан в simpleValue вместо xs:pattern. Возможно, ошибка в IDS-файле.
❌
Сущности не найдены (красный)
Тип сущности из IDS не существует в IFC4. Проверьте IDS-файл.
🚫
IFC не будет сгенерирован (красный)
Все PSet описаны regex — генерация невозможна.

Что делать, если что-то не работает

🐛
«Ошибка парсинга» в логе

Проверьте, что файл действительно в формате IDS (XML). Откройте его в текстовом редакторе — должен быть тег <ids:specification>.
